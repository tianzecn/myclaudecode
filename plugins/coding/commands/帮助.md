  ## Coding 插件 v1.0.0

  **AI 驱动的需求分析与 GitHub Issue 自动创建。**

  自动分析项目代码库、文档和配置，生成结构化、可执行的 GitHub Issue，包含技术建议和验收标准。

  ---

  ## 可用命令（3 个）

  | 命令 | 说明 |
  |------|------|
  | **/coding:需求分析** | 分析项目并根据需求描述创建结构化的 GitHub Issue |
  | **/coding:开发归档** | 基于 Issue 完成开发归档 - 提交、推送、评论、关闭 |
  | **/coding:帮助** | 显示此帮助信息 |

  ---

  ## /需求分析 工作流程

  ### 阶段 1：项目上下文分析

  1. **读取文档** - README、CHANGELOG、docs/、CONTRIBUTING
  2. **分析配置** - package.json、tsconfig、docker、数据库配置
  3. **分析核心代码** - 路由、模型、服务、组件、状态管理

  ### 阶段 2：生成 GitHub Issue

  1. **组织内容结构**：
     - 标题（简洁，最多 50 字符）
     - 问题描述 / 需求背景
     - 预期目标
     - 技术方案建议
     - 验收标准
     - 相关资源

  2. **添加标签**：enhancement、bug、feature、priority、area

  3. **创建 Issue**：使用 `gh` CLI

  ### 阶段 3：返回结果

  - Issue 链接（可点击的 URL）
  - Issue 编号（#123）

  ---

  ## /开发归档 工作流程

  ### 阶段 1：提交前验证

  1. **检查 Git 状态** - 确认修改的文件和变更
  2. **获取 Issue 详情** - 获取 Issue 内容作为参考

  ### 阶段 2：生成开发总结

  1. **分析代码变更** - 审查实际修改
  2. **组织总结结构**：
     - 原始问题回顾
     - 技术实现方案
     - 关键代码变更
     - 遇到的问题和解决方案
     - 测试验证结果

  ### 阶段 3：Git 提交和推送

  1. **暂存变更** - `git add -A`
  2. **创建提交** - Conventional Commits 格式（中文）+ `Closes #X`
  3. **推送到远程** - 推送到当前分支

  ### 阶段 4：Issue 归档

  1. **添加完成评论** - 包含提交链接的完整总结
  2. **自动关闭 Issue** - 通过提交信息中的 `Closes #X`

  ### 阶段 5：返回结果

  - Issue 链接
  - 提交链接
  - 执行摘要

  ## 使用示例

  ### 功能请求

  ```bash
  /coding:需求分析 添加支持 OAuth2 的用户认证功能
  ```

  **结果**：创建 Issue #42，包含：
  - 现有认证模式的技术分析
  - OAuth token 的数据库迁移
  - API 端点规范
  - 带测试用例的验收标准

  ### Bug 报告

  ```bash
  /coding:需求分析 用户反馈登录 30 分钟后失败
  ```

  **结果**：创建 Issue #43，包含：
  - 根本原因分析
  - 会话超时调查
  - 带代码引用的修复建议
  - 基于时间的验收标准

  ### 功能增强

  ```bash
  /coding:需求分析 为仪表盘实现深色模式
  ```

  **结果**：创建 Issue，包含：
  - UI 组件分析
  - CSS/主题架构
  - 状态管理方案
  - 视觉验收标准

  ### 完整开发周期

  ```bash
  /coding:开发归档 42
  ```

  **结果**：完成 Issue #42，包括：
  - Git 状态验证
  - 开发总结生成
  - 带 "Closes #42" 尾注的提交
  - 推送到远程
  - Issue 上的完成评论

  ## 前置要求

  1. **GitHub CLI (gh)**：必须安装并认证

  ```bash
  # 安装
  brew install gh

  # 认证
  gh auth login
  ```

  2. **Git 仓库**：必须在带有 GitHub 远程的 git 仓库中运行

  ## Issue 结构

  ```markdown
  # [类型] 简短描述

  ## 问题描述 / 需求背景
  [为什么需要这个功能]

  ## 预期目标
  [完成的定义]

  ## 技术方案建议

  ### 后端
  [API、数据库、业务逻辑]

  ### 前端
  [组件、状态、UI]

  ### 技术栈
  [检测到的技术]

  ### 数据模型
  [表、字段、关系]

  ## 验收标准
  - [ ] 可衡量的标准 1
  - [ ] 可衡量的标准 2

  ## 相关资源
  [文档、代码路径、链接]
  ```

  ## 错误处理

  | 场景 | 解决方案 |
  |------|----------|
  | gh CLI 未安装 | 引导使用 `brew install gh` 安装 |
  | 未认证 | 引导运行 `gh auth login` |
  | 不在 git 仓库中 | 提示用户在 git 仓库中运行 |
  | 没有 GitHub 远程 | 请用户指定仓库 |
  | 需求不清晰 | 提出澄清问题 |
