## 人是怎么买东西的



在动手之前，咱们先想清楚一个问题： **如果是人来干这件事，流程是什么？**



这很重要， **因为让 AI 做事，本质上就是把人的工作流程翻译成 AI 能理解的步骤。**



人买东西的流程大概是这样：



第一步，打开 B 站，登录账号。



第二步，搜索「加湿器测评」，看看哪些视频播放量高、标题看起来靠谱、UP 主是不是专业做测评的。



第三步，点开几个视频，完整看完，边看边记笔记：这款容量多大、那款噪音怎么样、哪个品牌容易坏……



第四步，综合自己的需求（比如房间大小、预算、有没有特殊功能需求）和视频里学到的信息，筛选出几个备选项。



第五步，打开某宝、某东去比价，看看哪家店在做活动，最后下单。



**整个流程，核心就是：搜索 → 看视频 → 学习信息 → 做决策。**



那让 AI 来做，流程也应该是类似的。



但有个关键问题： **AI 不能直接看视频。**



虽然目前的 Gemini 确实有看视频的能力，但效果一般，而且一个视频占用 token 太多了，也无法做到一次性「看完」好几个视频。



![](images/Claude_Ski_5e0061bcf322eff30c1b050cf02e64_20251117_173558_330.png)



AI 能理解的是文字，你给它一个视频文件，它打不开，就算有些 AI 说能「看视频」，实际上也是先把视频转成文字描述，再理解内容。



所以， **AI 要「看」视频，得先把视频的内容转成文字，也就是字幕。**



这就是第一步要解决的问题：怎么让 AI 获取到 B 站视频的字幕？



## 写个脚本比打开浏览器更优雅



你可能会想：那就让 AI 操控浏览器，打开 B 站，搜索视频，然后用第三方插件获取字幕不就行了？



这个思路没错，技术上确实可行，但有几个问题：



**问题一：信息太多了**



打开浏览器访问 B 站，页面上的东西太杂了，广告、推荐视频、评论、标题、侧边栏……这些东西对人来说可以自动忽略，但对 AI 来说都是「信息」。



![](images/Claude_Ski_f33afab4add3a030aa49eae8f42a90_20251117_173558_330.png)



AI 会把整个页面的内容都读进去，这会让它的上下文窗口（理解成 AI 的「记忆容量」）快速膨胀，造成上下文瘫痪。



此时，关键信息被稀释，随着上下文的增长，模型的注意力会被分散。



就像你让一个人记住一整页报纸的所有文字，包括广告和页码，他当然记得很辛苦。



**问题二：需要占用我的电脑**



让 AI 操控浏览器，意味着它要在我的电脑上打开一个浏览器窗口，这个过程我没法在后台运行，得一直等着它操作完。



而且，这个浏览器窗口是我的电脑在跑，AI 本身没有自己的浏览器环境。



**所以，更好的办法是：直接写个脚本去获取字幕.**



B 站的视频字幕，其实是存在 B 站服务器上的一个文件。



我们可以写一个简单 Python 脚本，直接调用 B 站的接口，把字幕文件下载下来。



![](images/Claude_Ski_dc075483e100bd8ac3f7e6ecdd68dc_20251117_173558_330.png)



这样做的好处是：

* 干净，脚本只获取字幕，不会加载页面上的其他内容。

* 快，不需要等浏览器加载、渲染页面。

* 静默，在后台运行，不占用我的屏幕。

*

看到这里要写编程你先别慌， 我们想一下：我得先写这个脚本吗？我得去研究 B 站的 API、怎么发请求、怎么解析字幕格式？



真不用，让 Claude Code 帮我写行了。

## Cookie 是什么



如果你不懂编程，可能不太清楚 Cookie 是什么，我简单解释一下。



**Cookie 是网站用来「记住你」的一串字符。**



你登录 B 站后，浏览器会保存一个 Cookie。



下次你再打开 B 站，浏览器会自动把这个 Cookie 发给 B 站的服务器。



B 站服务器一看：「哦，这个 Cookie 我认识，这是用户 xxx，已经登录过了。」然后就不会让你重新登录。



Cookie 就像一张「通行证」。有了它，网站就知道你是谁、你有没有登录、你有什么权限。



那为什么获取字幕需要 Cookie 呢？



因为 B 站有 **反爬虫机制，&#x20;**&#x5982;果一个请求没有 Cookie，B 站的服务器会认为：「这不是正常用户在看视频，这是个爬虫在抓数据。」然后就拒绝返回字幕。



但如果请求里带上了 Cookie，B 站就会认为：「这是一个已登录的用户在看视频。」然后就正常返回字幕。



所以，我得给 Claude Code 提供一个 Cookie。



## 怎么获取 Cookie



获取 Cookie 很简单：



第一步，打开浏览器，登录 B 站，我用的是自己的 B 站小号（不用大号是怕被封号，虽然这个风险很小）。



第二步，按 **F12&#x20;**&#x952E;，打开浏览器的「开发者工具」。



![](images/Claude_Ski_d89dc7e9fe04a0f2f0016639924b19_20251117_173558_331.png)



第三步，点击「Application」标签（应用），然后点击「Cookies」，找到bilibili 的页面，把这里的「name」和「value」的值复制粘贴下来。



![](images/Claude_Ski_d29e3d91d4805e907048abbc857bfc_20251117_173558_331.png)



**你不用管到底哪一段 Cookie 才是有用的，不需要你背后的原理。**



直接把所有都 input 给Claude，它会自己判断哪一段是有用的，Claude Code 把 Cookie 加到了代码里，再次运行。



这次成功了。



终端输出了字幕内容，我看了一眼，确实是视频里的字幕，没问题。



**到这里，第一步就完成了：我们有了一个能获取 B 站视频字幕的脚本。**



## 优化字幕格式



脚本能用了，但还有个问题。



我让 Claude Code 打开获取到的字幕文件看了一眼，发现是 SRT 格式，长这样：



```plaintext
1
00:00:01,000 --> 00:00:03,500
大家好，今天测评加湿器

2
00:00:03,500 --> 00:00:06,000
我们准备了五款不同价位的产品

3
00:00:06,000 --> 00:00:09,000
首先看第一款，某品牌的入门级产品
```



每一段字幕前面都有序号，还有时间戳（00:00:01,000 这种）。



这些信息对人看视频有用，你知道这句话出现在视频的第几秒，可以快速跳转，但对 AI 来说，这些就是噪音。



AI 要理解的是字幕的内容，序号和时间戳不仅没用，还会占用 token。



这就是上下文工程的一个关键原则： **确保给 AI 的输入信息密度是高的。**



去掉无效信息，有两个好处：

* **节省成本&#x20;**。每个 token 都要花钱，无效信息越多，成本越高。

* **提高质量&#x20;**。AI 的上下文窗口是有限的，塞了太多无效信息，真正重要的内容就被稀释了。

*

所以我让 Claude Code 修改脚本，把 SRT 格式转成 **纯文本的 Markdown 格式&#x20;**，删掉时间戳和序号，只保留字幕内容：



```plaintext
我需要你把字幕转成 Markdown 格式，修改时间戳格式、删掉字幕序号，只保留文字内容，这样更适合AI阅读。
```



Claude Code 很快改好了，现在获取到的字幕是这样的：



```markdown
00:01 大家好，今天测评加湿器

00:03 我们准备了五款不同价位的产品

00:06 首先看第一款，某品牌的入门级产品
```



干净、清爽、信息密度高。

## AI 真的能看懂字幕吗



现在脚本能用了，字幕格式也优化了，接下来要验证一个核心问题：



**AI 读完这些字幕后，真的能给出靠谱的选购建议吗？**



这个问题很关键，如果 AI 读完字幕还是一头雾水，或者给的建议不靠谱，那整套流程就没意义了。



所以我在当前对话里直接测试。



我让 Claude Code 用刚才写的脚本，获取几个加湿器测评视频的字幕，然后我跟它说：



```plaintext
基于这些字幕内容，给我一份选购建议。
```



Claude Code 开始工作，它读完字幕后，输出了一份报告，内容大概是：



* 根据房间面积，XX 平米以下选 XX 升，XX 平米以上选 XX 升

* 某几个品牌的噪音表现好，适合放卧室

* 有些品牌的水箱设计不合理，容易漏水

* XX 品牌和 XX 品牌价格相近，但 XX 品牌功能更多



![](images/Claude_Ski_ab75ea85ff4e94d23ae9d8fd488c16_20251117_173558_331.png)



我看完之后，发现这个建议是靠谱的。



AI 确实能从字幕里提取出关键信息：哪些功能重要、哪些品牌性价比高、要避开哪些坑，然后给出结构化的建议。



而且我发现一个有意思的点： **我没有写复杂的提示词，AI 就能理解任务。**



我只是说「基于这些字幕给我选购建议」，AI 就知道要总结优缺点、对比不同产品、考虑我的需求。



可能是因为「看测评然后给选购建议」这个任务本身很符合人类的直觉，AI 也能自然地理解。



到这一步，其实在这个对话窗口里，我已经能让 AI 帮我完成选购决策了。



**但问题是：下次我想买别的东西，还得重复这一遍流程。**



我得重新解释任务、重新让 AI 调用脚本、重新交代要求，就像每次都要培训一个新实习生，很累。



这就是 **Claude Skills&#x20;**&#x8981;解决的问题。



## Skills 就是把流程固化下来



Claude Skills 是什么？



简单说，就是 **提前准备好的「技能包」。**



![](images/Claude_Ski_c2f5bff12e0ef1a540f71b3e7796bc_20251117_173558_331.png)



一个 Skills 里可以包含：

* **提示词&#x20;**：告诉 AI 怎么干活

* **参考文档&#x20;**：比如操作规范、格式要求

* **代码脚本&#x20;**：让 AI 直接调用，不用每次重新写

*

你把这些东西打包成一个文件夹，给到 AI ，以后每次对话，AI 都能自动调用这个「技能」，不用你重复解释。



**AI 会自己判断什么时候该用哪个 Skills。**



你不用输入什么指令，只要说「帮我看看加湿器的测评」，AI 看到关键词，自己就知道调用「B 站测评分析」这个 Skills。



所以接下来，我要把刚才这套流程做成一个 Skills。

Claude Code 完全有能力做这件事。

## 检查 AI 写的 Skills 内容



接下来我要检查一下 AI 写的内容，看看有没有问题，毕竟现在的 AI 真就做不到「左脚踩右脚登天」，需要人类介入干预才行。



先看 `SKILL.md` ，这个文件定义了 Skills 的信息，最开头写的是Skills的简介：



```yaml
---
```

这个 description（描述）很重要。它决定了 AI 什么时候会触发这个 Skills。



比如我说「帮我看看加湿器的 B 站测评」，AI 看到「B 站」「测评」这些关键词，就会判断：「哦，这个任务需要用 bilibili-subtitle 这个 Skills。」然后自动调用。



我看了一下，描述写得挺好，关键词都覆盖到了。



接着看下面的提示词，这是 Skills 的核心文件，AI 写的内容大概是：



```markdown
# B站视频字幕获取与分析

1. 搜索B站视频

3. 分析字幕内容

## 使用方法

## 注意事项

- 需要提供B站的 Cookie
- 字幕会自动转换为 Markdown 格式
- 适合批量数据处理，比如收集学习资料、分析热门话题、训练NLP模型、内容研究
```



前面部分都没问题。但我发现最后一段有点问题：



> 适合批量数据处理，比如收集学习资料、分析热门话题、训练 NLP 模型、内容研究

## 发现问题，继续迭代



不过，我又发现了一个问题。



**Claude Code 这次只获取了 10 个视频的字幕，而且没有显示视频的播放量。**



10 个视频太少了，可能会漏掉一些优质内容，而且没有播放量信息，AI 没法判断哪个视频更值得「看」，播放量高的视频，通常质量也更好。



所以我让它修改脚本：



```plaintext
改一下脚本，一次性获取 30 个视频，并且获取每个视频的播放量。然后基于播放量和标题，你自己判断应该「看」哪几个视频（从 30 个里筛选最值得看的 6 个），再给我建议。
```



![](images/Claude_Ski_dc6293f9667c1e3d42f283c1c48a70_20251117_173619_860.png)



为什么是 6 个？因为太多了 AI 读不完（上下文会爆），太少了信息又不够全面。



6 个视频的字幕，大概在 AI 的处理范围内，又能覆盖不同角度的测评。



Claude Code 修改了脚本，添加了获取播放量的逻辑， **还加了筛选机制：综合考虑播放量、时长、内容专业性。**



![](images/Claude_Ski_dc075483e100bd8ac3f7e6ecdd68dc_20251117_173619_860.png)



这次测试，我让它帮我看空气加湿器的选购建议：



```plaintext
我想买一个空气加湿器，找一下这方面的测评视频，然后转成字幕看一下UP主怎么说的（从30个里筛选最值得看的6个），然后最终给我一个选购建议（包含溯源视频链接和观看时间轴）。
```



Claude Code 开始工作：



1. **搜索视频&#x20;**：在 B 站搜索「空气加湿器测评」，获取到 30 个视频的信息（标题、播放量、链接）

2. **筛选视频&#x20;**：根据播放量和标题关键词，筛选出 6 个最值得看的视频

3. **获取字幕&#x20;**：逐个获取这 6 个视频的字幕，转成 Markdown 格式

4. **分析内容&#x20;**：读完字幕，提取关键信息

5. **输出报告&#x20;**：生成一份选购建议



最终输出的报告长这样：



![](images/Claude_Ski_ab75ea85ff4e94d23ae9d8fd488c16_20251117_173619_861.png)



我看完之后，非常满意。



关键信息都有了，该怎么选、要避开什么坑、具体推荐哪几款。



还有溯源链接，如果我想确认某个信息，可以点开视频看原片，时间轴都标好了，直接跳到重点部分。



整个过程，从我提需求到拿到报告，大概 10 分钟。



如果我自己去完成这个流程，边看边记录，至少要 1 小时。

AI 会自己去 B 站找测评、读字幕、总结信息、给建议，我只需要看最终的报告，然后做决策。

## Skills 还会继续进化



当然，现在这个版本还不是完美的。



后面我可能还会继续迭代：



**优化提示词&#x20;**：让 AI 的分析更精准，比如自动识别软广、重点关注技术细节。



**优化脚本&#x20;**：提高获取字幕的成功率，支持更多视频平台（比如抖音、YouTube），自动处理没有字幕的视频（调用语音识别 API 生成字幕）。



**增加交互&#x20;**：让 AI 在分析完之后主动问我：「你的预算是多少？房间多大？有没有特殊需求？」然后根据我的回答进一步筛选推荐。



这就是 Skills 的好处： **做成一个初版后，可以不断测试、不断迭代，慢慢打磨成更好用的版本。**



就像写提示词一样。你不可能一次写出完美的提示词，都是先写个初版，然后通过测试发现问题，再改进。



Skills 也是一样的逻辑。

## Claude Skills 的价值



有人说，Skills 的价值比 MCP 还大。也有人说，这不就是 MCP 的另一种形式吗。我是这样看的。



过去我们怎么传递 AI 的使用经验？



**通过分享提示词、分享对话流程。**



比如大神们在互联网上分享了很多优质提示词，其他人看到了，复制过来用，就不需要重复造轮子，直接「站在巨人的肩膀上」。



![](images/Claude_Ski_854d975b2791aa9ac0209124e0a942_20251117_173619_861.png)



人类文明之所以能进步，就是因为有了文字，文字让知识得以记录、传播、积累。



你不需要自己重新发明轮子，你可以学习前人的经验，然后在此基础上继续创新。



AI 时代也一样。



以前，我们只能传递「怎么跟 AI 对话」（提示词）。现在有了 Skills， **我们不仅能传递知识，还能传递工具。**



你可以把一套完整的工作流程，包括提示词、参考资料、代码脚本，打包成一个 Skills，分享给其他人。



其他人拿到之后，不需要理解底层原理，不需要自己写代码，直接就能用。



**就像开源社区的代码一样。**



大家把自己写的 Skills 开源出来，互相学习、互相改进。



有人发现了一个 bug，提交一个修改建议。有人想到了新功能，加进去，大家一起用。



这样一来， **群体智慧能够变成 AI 的智慧。**



![](images/Claude_Ski_01dad095ab3a190a5e8f499adddaf5_20251117_173619_861.png)



AI 原本是个「好像什么都会，但好像又不太会」的实习生。



有了 Skills 这个机制，它终于有了一套系统化的学习方式： **不只是记住知识，还能记住工具、记住方法论、记住最佳实践。**



而这些 Skills，是全人类共同创造的，每个人贡献一点点，AI 就能变得越来越强大。



你可能注意到了一个细节：这一次我让 Claude Code 写B站视频脚本的时候，它会先去 **GitHub 上搜有没有人已经写过了&#x20;**。



然后它找到了别人写好的库，直接拿来参考，或者调用那个库。



**这就是在找「最佳实践」。**



相当于 AI 知道：「这个功能别人已经做过了，而且做得很好，我不需要从零开始写，直接用别人验证过的方案就行。」



![](images/Claude_Ski_0b1e30e95e644a4228bb75948833bf_20251117_173619_861.png)



**未来，使用 AI 也应该有类似这样的最佳实践。**



哪怕再过几年，真的出现了无所不能的 AGI，AGI 有了这些最佳实践之后，在执行任务的时候也能更快、更准、更稳定。



## 即使有了 AGI，Skills 仍然重要



更关键的是， **有些隐性知识不分享出来，AI 是不可能学会的。**



比如你或者你团队的内部流程。



AI 在训练的时候，没有你们内部流程的数据。



哪怕有了 AGI，它也不可能直接在不研究的情况下，弄懂你们公司的工作流程，然后产出你满意的结果。



也就是说， **哪怕有了 AGI，也仍然需要准备好这么一份 AI 能看得懂的「技能包」&#x20;**，让它看懂你的工作流，上手才能更快、做得才能更好。



就像新员工入职，再聪明的人也得先看公司的操作手册、熟悉内部系统、了解业务流程，AGI 也一样。



所以，到了这个阶段，我认为最重要的就是：



**把自己工作中的所有流程，尽可能地都转换成「AI 能理解的步骤」，也许是变成一个个「技能包」，也许是变成一套基于 n8n 的 AI 工作流，都可以。**



![](images/Claude_Ski_6cbd3aa7531527bc5ba28c3e2c3021_20251117_173619_861.png)



这不是在为某个具体的 AI 产品做准备，而是在为「AI 深度参与工作」的未来做准备。



Skills 只是一个开始，未来可能会有更多类似的机制，让我们能更好地「训练」AI，让 AI 更好地理解我们的工作方式。



但核心逻辑不会变： **把隐性知识显性化，把工作流程结构化，让 AI 能够学习、复用、协作。**



这才是 Skills 最让我兴奋的地方。
